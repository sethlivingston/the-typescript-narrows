## Implementation Decisions

### prefer-for-of
- Use `for...of` over `.forEach`. forEach swallows break/continue/return/await.
- Address the closure gotcha (loop variable scoping with `const` in for...of is safe since each iteration gets a fresh binding).
- Severity: [M], Enforcement: skill-only
- Topic group: new "Iteration and Transforms" or fit into existing group

### prefer-early-return
- Standalone opinion. Fail fast, keep happy path left-aligned (Go-style guard clauses).
- Reduces nesting, improves readability.
- Severity: [M], Enforcement: skill-only
- Topic group: new "Control Flow" or standalone

### prefer-arrow-functions
- Arrow functions by default. Function declarations only when hoisting is needed (mutual recursion, top-level organization before definition).
- Severity: [S], Enforcement: skill-only
- Topic group: fits "Style" or new "Functions" group

### explicit-return-types
- Require explicit return types on exported functions for API stability.
- Internal/private functions can rely on inference.
- Severity: [M], Enforcement: both (lint-enforceable via @typescript-eslint/explicit-function-return-type)
- Topic group: fits "Type Safety" or "Type Declarations"

### prefer-undefined
- Pick `undefined` over `null`. Matches JS defaults (missing params, void returns, optional properties).
- Exception: when interfacing with APIs that use null explicitly (JSON, DOM).
- Severity: [S], Enforcement: skill-only
- Topic group: fits "Null Handling & Narrowing"

### no-reduce
- Ban `.reduce` entirely. Use `for...of` or `.map`/`.filter` combinations.
- No "trivial case" exception — zero ambiguity.
- Severity: [M], Enforcement: skill-only
- Topic group: same group as prefer-for-of (Iteration and Transforms)

### Async/Await
- Already covered by existing prefer-async-await opinion. No changes needed.
