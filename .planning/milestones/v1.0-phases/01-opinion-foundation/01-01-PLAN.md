Create the directory `docs/opinions/` and the file `docs/opinions/README.md` with the following content structure:

1. Title: `# The TypeScript Narrows - Opinions`

2. `## Philosophy` section:
   - Core message: This corpus provides a single, well-reasoned opinion for every common TypeScript decision point. When two reasonable developers would do it differently, we pick a side and explain why.
   - Mention that opinions are self-contained -- no external links.
   - Mention the inclusion bar: "prevents a real, recurring problem" or "eliminates a common ambiguity."

3. `## How Opinions Are Structured` section:
   - Explain the two parts: YAML frontmatter (machine-readable metadata) and markdown body (human-readable content).
   - List frontmatter fields with descriptions:
     - `id`: Slug-based identifier matching the filename
     - `title`: Short imperative title
     - `severity`: One of `bug-prevention`, `maintenance`, `style`
     - `enforcement`: One of `skill-only`, `lint-enforceable`, `both`
     - `confidence`: One of `strong`, `moderate`
     - `tags`: Freeform array of topic tags
     - `related`: Optional array of related opinion IDs
     - `lint`: Required if enforcement is not `skill-only`. Contains `type` (`existing` or `custom`) and `rule` (if type is `existing`)
   - List body sections: Stance, Why, Do (one or more), Don't (one or more), Exceptions (optional)
   - Note that code examples are minimal 3-8 line snippets; opinions may include multiple Do and/or Don't blocks when the pattern has distinct variations.

4. `## Severity Tiers` section with a table:
   | Tier | Meaning |
   |------|---------|
   | bug-prevention | Can cause runtime errors, data corruption, or silent wrong behavior |
   | maintenance | Makes code harder to understand, refactor, or extend over time |
   | style | Consistency and readability with no functional impact |

5. `## Global Exception Clause` section:
   - All opinions allow exceptions when there is no other viable alternative.
   - If you must deviate, document why in a code comment.
   - The goal is consistency, not dogma.

6. `## For AI Agents` section:
   - When consulted about a TypeScript decision point, provide the opinion's stance and rationale.
   - If the developer asks why, explain using the Why section.
   - If they push back with a valid exception, acknowledge it -- these opinions are strong defaults, not absolute rules.
   - The frontmatter is machine-readable -- parse it for severity, enforcement type, and related opinions.
