**Researched:** 2026-03-17
**Domain:** Traceability validation, npm publishing, Claude skill deployment

## Summary

Phase 5 closes the loop on the project by validating that all 59 opinions are covered by the appropriate artifact(s) and then publishing both artifacts. The work breaks into three streams: (1) build a traceability script that reads opinion frontmatter and cross-references skill reference files and ESLint config rules, producing a markdown matrix and exit-code validation; (2) prepare the ESLint plugin for npm publish (version bump, metadata, LICENSE, README); (3) prepare the skill for deployment (README, LICENSE, verify registry compatibility).

The codebase is well-structured for automation. Opinion files have consistent YAML frontmatter with `id`, `enforcement`, and `lint` fields. The strict config in `eslint-plugin/src/configs/strict.ts` enumerates all active rules. Skill reference files in `skill/the-typescript-narrows/references/` map cleanly to opinion topics. The traceability script can parse these three sources and produce a complete coverage matrix.

**Primary recommendation:** Build the traceability script first (it validates coverage and surfaces any gaps), resolve gaps, then prepare publishing artifacts and publish.

**Recommendation:** Use gray-matter for frontmatter parsing. It handles the `---` delimiter extraction cleanly and is the de facto standard for markdown frontmatter in the Node.js ecosystem.

**Installation (in eslint-plugin or root scripts dir):**
```bash
npm install --save-dev gray-matter
```

Note: Since there is no root package.json, the traceability script should either be a standalone Node.js file using dynamic imports/built-in modules only, or gray-matter should be added to the eslint-plugin devDependencies (since that is the only package.json). Alternatively, the script can split on `---` manually and use js-yaml (already available as a transitive dep). The manual approach avoids adding a new dependency.

**Recommended: Zero-dependency approach.** Parse frontmatter with a simple regex split on `---` and use the built-in Node.js `fs` module. The YAML in opinion files is simple enough (flat key-value with one nested object) that a minimal parser or even regex extraction suffices. This avoids adding dependencies to a project that has no root package.json.

### Pattern 1: Traceability Script Design
**What:** A Node.js ESM script that reads all 59 opinion markdown files, extracts frontmatter, and cross-references against the strict config rules and skill reference files.
**When to use:** Run before publish to validate coverage; exit non-zero on gaps.

**Data flow:**
1. Read `docs/opinions/*.md` (excluding INDEX.md and README.md) -- extract `id`, `enforcement`, `lint.rule`, `lint.type`
2. Read `eslint-plugin/src/configs/strict.ts` -- extract all rule names from the `rules:` objects (regex parse, not TS compilation)
3. Read `skill/the-typescript-narrows/SKILL.md` and `references/*.md` -- check for opinion mentions
4. Cross-reference: for each opinion, check if the expected artifacts exist
5. Output `docs/TRACEABILITY.md` and exit with code based on gap count

**Example (frontmatter extraction):**
```javascript
// scripts/generate-traceability.mjs
import { readFileSync, readdirSync, writeFileSync } from 'node:fs';
import { join } from 'node:path';

function parseFrontmatter(content) {
  const match = content.match(/^---\n([\s\S]*?)\n---/);
  if (!match) return null;
  // Simple YAML parsing for flat structure
  const yaml = {};
  const lines = match[1].split('\n');
  let currentKey = null;
  for (const line of lines) {
    const kvMatch = line.match(/^(\w+):\s*(.+)/);
    if (kvMatch) {
      const [, key, value] = kvMatch;
      yaml[key] = value.replace(/^["']|["']$/g, '');
      currentKey = key;
    } else if (line.match(/^\s+\w+:/) && currentKey === 'lint') {
      const nested = line.match(/^\s+(\w+):\s*(.+)/);
      if (nested) {
        yaml.lint = yaml.lint || {};
        if (typeof yaml.lint === 'string') yaml.lint = {};
        yaml.lint[nested[1]] = nested[2].replace(/^["']|["']$/g, '');
      }
    }
  }
  return yaml;
}
```

### Pattern 2: Strict Config Rule Extraction
**What:** Parse `strict.ts` to extract all configured rule names without requiring TypeScript compilation.
**Why:** The strict config is TypeScript source; regex extraction avoids needing to build first.

```javascript
// Extract rule names from strict.ts
function extractConfiguredRules(strictTsContent) {
  const rules = new Set();
  // Match rule names in quotes followed by colon
  const rulePattern = /'([^']+)':\s*[[\]'"]|"([^"]+)":\s*[[\]'"]/g;
  let match;
  while ((match = rulePattern.exec(strictTsContent))) {
    rules.add(match[1] || match[2]);
  }
  return rules;
}
```

### Pattern 3: Skill Coverage Check
**What:** Verify each opinion has a corresponding mention in skill reference files.
**Why:** Skill-only opinions must have reference file coverage; this validates COVR-01.

The skill reference files are organized by topic (type-safety.md, error-handling.md, etc.). Each opinion's `id` should appear as a concept in the corresponding reference file. The check can verify that the opinion's title or stance text appears in at least one reference file.

### Anti-Patterns to Avoid
- **Importing strict.ts for rule extraction:** Do not `import()` the TypeScript source -- it requires compilation. Parse the text instead.
- **Hardcoding opinion lists:** The script must dynamically read `docs/opinions/` so new opinions are automatically included.
- **Publishing without building:** Always run `npm run build` before `npm publish`. Use `prepublishOnly` to enforce this.

## Common Pitfalls

### Pitfall 1: Version Mismatch Between package.json and Plugin Meta
**What goes wrong:** `eslint-plugin/src/index.ts` has `version: '0.1.0'` hardcoded in the plugin `meta` object. If you bump package.json to 0.9.0 but forget this, the plugin reports the wrong version.
**Why it happens:** Version is duplicated in two places.
**How to avoid:** Update BOTH `eslint-plugin/package.json` version AND `eslint-plugin/src/index.ts` meta.version in the same task.
**Warning signs:** `npm pack --dry-run` shows the right version but ESLint reports the old one.

### Pitfall 2: Missing `files` Field Includes Too Much or Too Little
**What goes wrong:** The current `"files": ["dist"]` is correct for code, but README and LICENSE need to be included too.
**Why it happens:** npm automatically includes package.json, README, and LICENSE at the package root, so these ARE included without explicit listing.
**How to avoid:** Run `npm pack --dry-run` after adding README and LICENSE to verify they appear in the tarball.
**Warning signs:** Published package missing docs.

### Pitfall 3: Strict Config Rule Name Mismatch
**What goes wrong:** Traceability script reports a "gap" because the opinion's `lint.rule` field uses a different name format than what's in strict.ts.
**Why it happens:** Some opinions reference the rule differently (e.g., with or without `@typescript-eslint/` prefix).
**How to avoid:** Normalize rule names during comparison. The opinion frontmatter `lint.rule` field already includes the full prefixed name (e.g., `@typescript-eslint/no-explicit-any`).

### Pitfall 4: npm Publish Without Build
**What goes wrong:** Publishing includes stale dist/ files from a previous build.
**How to avoid:** Add `"prepublishOnly": "npm run build && npm run test"` to package.json scripts.

### Pitfall 5: Skill README Duplicating SKILL.md Content
**What goes wrong:** Two files go out of sync.
**How to avoid:** README focuses only on: what this is, how to install, link to SKILL.md. No opinion content in README.
